# Phase 3 — Portal gap-fill (frontend)

Branch: `feature/portal-gap-fill`

This task wired the frontend to a batch of backend additions made to close gaps the
UCT / UP / Wits / UJ portals exposed. The backend shipped new profile fields, a new
`/contacts` resource, multiple programme choices on applications, an NSC level per
subject, and a new document type. This doc covers what changed on `uniflo-web`.

## Starting point

The contract is owned by `uniflo-api` and consumed via `lib/api/schema.d.ts`
(generated by `npm run types:api` from the live OpenAPI spec — never hand-written,
per `AGENTS.md`). The first step was regenerating that file once Partner B deployed
the new spec, then building everything on top of the generated types.

## What was built

### 1. Documents — `GRADE11_RESULTS`

- `lib/constants/documents.ts`: added `OPTIONAL_DOC_TYPES` (`GRADE11_RESULTS`) and
  extended `DOC_LABELS`. `REQUIRED_DOC_TYPES` is **unchanged** — Grade 11 is a UP
  alternative, not a gating requirement.
- `components/documents/upload-form.tsx`: split zones into
  `REQUIRED_ZONE_CONFIGS` + `OPTIONAL_ZONE_CONFIGS`. The progress bar / "all
  uploaded" CTA counts required zones only; the optional Grade 11 zone renders
  under its own "Optional" heading.

### 2. Academic records — NSC level per subject

- `components/academic-records/records-form.tsx`: each subject row gained an
  optional `nscLevel` (1–7) select beside the required mark. Validated only when
  set; folded into the subject payload as `nsc_level` (omitted when blank).
- The review screen subject chips show `· L<level>` when present.

### 3. Applications — multiple programme choices

- `lib/state/selection.tsx`: `SelectionEntry` gained `additionalProgrammes?:
  string[]` (the `UPDATE` patch type was widened to match).
- `components/applications/application-fieldset.tsx`: primary programme + up to
  two removable additional choices (`MAX_ADDITIONAL_PROGRAMMES = 2`, total ≤ 3).
- `components/applications/new-applications-form.tsx`: per-slot validation
  (each choice 3–120 chars), context write-back including `additionalProgrammes`.
- `components/applications/review-screen.tsx`: POST body includes
  `additional_programmes` when present; the "Your applications" rows list the
  extra choices.
- `components/applications/application-detail.tsx`: renders the ordered
  `choices[]` with per-choice eligibility (`Eligible` / `Not eligible` /
  `Pending` — eligibility is filled in later by the automation).
- `components/applications/application-list.tsx`: a "+N more choices" indicator.

### 4. Profile — optional Phase-3 fields

All new fields are **optional** and deliberately **excluded from the
profile-completeness gate**. They live in the edit form (`/profile/edit`) and
the read-only overview, **not** in the setup wizard — onboarding stays
required-only so the flow doesn't regress.

- `lib/constants/profile-enums.ts`: `TITLE_OPTIONS`, `CURRENT_ACTIVITY_OPTIONS`.
- `components/profile/edit-form.tsx`: new sections — Additional names
  (title / middle / maiden / preferred), SA-citizen flag, Mailing address
  (shown only when it differs from residential), disability detail + assistance
  (shown when a disability is declared), Studies & activity, Funding & residence,
  NBT (reference / year / date), and a free-form Redress key/value editor (UCT).
- `components/profile/overview.tsx`: an "Additional details" section that renders
  only the populated optional fields (avoids a wall of em-dashes).
- New primitives: `components/ui/checkbox.tsx`, `components/ui/textarea.tsx`.

### 5. Contacts — new resource

- `lib/api/contacts.ts`: typed `listContacts` / `upsertContact` /
  `deleteContact` helpers + `CONTACT_TYPES` copy.
- `components/contacts/contacts-manager.tsx`: one card per type
  (`next_of_kin` / `fee_payer` / `guardian` / `emergency`), each saved and
  removed independently. Inline validation covers email format, 4-digit postal
  code, and the **Wits rule** (next_of_kin email & mobile must differ from the
  applicant — applicant email comes from the auth session, phone from the
  profile). The **UJ fee-payer** full-address need is surfaced as card copy
  (not hard-blocked, since the backend accepts a partial address).
- New route `app/(app)/contacts/page.tsx` + a "Contacts" sidebar entry.

## Design decisions

- **Optional-everywhere for the new profile fields.** They feed automation but
  must never block completeness or onboarding. Edit form + overview only.
- **Match the locked design system** rather than introduce new components — the
  brand identity is fixed, so the two new primitives (Checkbox, Textarea) mirror
  Input/Select exactly.
- **`additional_programmes` only sent when non-empty**, and trimmed/filtered on
  write-back, so a student who lists one programme sends the same payload as
  before.
- **Contacts cleaned to `null` on save** (empty string → null) so the backend
  stores blanks consistently.

## Cross-portal review follow-up (uniflo-api PR #49)

After Partner B live-verified all four portals, a second batch of changes
landed on this same branch:

### Contacts collapsed to one section

UP asks for no contact person; UCT, UJ and Wits each derive next-of-kin /
emergency / fee-payer roles from one person, and the backend now resolves all
portal roles from any captured contact via fallback chains. So the four-card
contacts page became:

- One **"Parent / Guardian"** card, saved as `contact_type: "guardian"`.
- A **"Someone different pays the fees"** toggle that reveals a `fee_payer`
  card only when on. Switching it off deletes a saved fee_payer server-side
  (with revert-on-failure so the error stays visible).
- Next-of-kin and emergency forms are gone. Legacy rows of those types are
  ignored on load — the backend reuses the guardian for those roles.
- The Wits must-differ rules (email + mobile vs the applicant's) moved to the
  guardian. The guardian's SA ID number is now required to save the card at
  all, because UCT demands it once any guardian detail is given (13-digit
  format enforced). Relationship became a select (Mother/Father/Guardian/…).
- Guardian address is optional — the backend falls back to the student's
  address where a portal demands one. The UJ full-address need stays a hint
  on the fee-payer card.

### Programme choices 2–3 are live

The extra choices now genuinely reach the portals (UP and UCT fill two, Wits
up to three; UJ's second choice is backend-pending). The apply form labels
them **"Second choice (recommended)"** and **"Third choice (Wits only)"**,
and the add button names the slot it would add.

### Automation gate on current activity

The automation refuses to run for students who aren't currently in Grade 12
(the run fails with `form_submit_failed`), so the apply flow blocks them up
front: `/applications/new` server-fetches the profile and shows an "apply
directly on the portal" notice instead of the form when `current_activity`
is Upgrading matric / Gap year / Employed / At university. The review screen
mirrors the check and disables submit (belt and braces — the selection
context makes review unreachable without passing the form, but it's the
component that POSTs). Unset and "Other" activities fail open, as does a
failed profile fetch. The list lives in `lib/constants/profile-enums.ts`
(`AUTOMATION_BLOCKED_ACTIVITIES`).

### Records page made config-driven

The Grade 11 / Grade 12 April sections now render from a `SECTIONS` array,
and the form treats every non-baseline type uniformly (load-on-mount, inline
success). Adding **Grade 12 June** is one `SECTIONS` entry plus a
`RECORD_TYPE_LABELS` entry once the backend deploys the enum value — see the
lag note below.

### Regenerated schema brought unrelated surface

`npm run types:api` also pulled in backend PR #29's surface: an
`action_required` application status, `pending_challenge` on applications,
and `/applications/{id}/consent`, `/challenge`, `/field-mappings` detail
endpoints. The status badge now renders `action_required` ("Action needed",
warning tone); **no challenge/consent UI has been built yet** — that's its
own task.

Profile fields the review flagged (religion, the mailing-address toggle,
`applying_institutional_funding`, `preferred_residence` behind
`wants_residence`) were all already captured — no change needed.

## Deviations / notes for the next person

- **The deployed spec lags uniflo-api PR #49 in two places** (as of
  2026-06-12): `RecordType` has no `grade_12_june`, and `CurrentActivityEnum`
  has no "Completed matric" value (which the partner message lists as
  blocked). When the backend deploys: regenerate types, add the June section
  + label, and add "Completed matric" to `AUTOMATION_BLOCKED_ACTIVITIES`.
- The contacts upsert is `POST /contacts` per the deployed spec (the partner
  message said PUT — the spec wins).
- `MATRIC_RESULTS` is still labelled "Grade 11 final results" in the existing
  copy, and the backend additionally has `GRADE11_RESULTS`. The naming overlap
  is a backend artefact — `GRADE11_RESULTS` is treated purely as the optional UP
  alternative here. Worth tidying the labels with Partner B at some point.
- `redress_factors` is modelled as free-form key/value rows folded into an
  object on save. If UCT's redress questions get a fixed schema later, this
  should become a structured form.
- **Browser verification is partial.** Public pages render with zero console
  errors and the production build compiles every route (incl. `/contacts`), but
  the new UI is auth-gated and a full Playwright walkthrough at desktop/tablet/
  mobile needs a test-student login + the live backend. That pass is still
  outstanding.
